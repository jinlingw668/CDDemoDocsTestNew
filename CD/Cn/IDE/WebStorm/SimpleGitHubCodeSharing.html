说明

  1. GitHub及其所基于的Git系统提供了功能强大且很复杂的源代码共享功能， 是目前软件开发业界最为流行的VCS系统，Version Control System。 该系统的核心是源代码的分支(branch)与合并(merge)。 但本网页只介绍最简单的不创建分支的GitHub源代码共享操作。 主要说明在多人合作开发中，将源代码从GitHub拉到本地(pull)和将本地代码推送(push)到GitHub的操作。
  2. 使用Git进行代码共享的基本步骤是：
    提交(Commit)→拉取(Pull)(或更新(Update))→冲突解决(Resolve Confilts)→再提交(Cmmit)→推送(Push)

提交(Commit)操作

  1. Commit操作简单地说就是在本地形成一个关于代码的阶段性快照。
  2. WebStorm中可以使用菜单(VCS→Commit)或(Ctrl+K快捷键)进行提交操作。
  3. 代码研发过程中，应经常执行Commit操作，以记录研发过程的阶段性，并为合作研发中的代码合并打下良好的基础。

拉取(Pull)操作及冲突解决

  1. Pull操作就是将GitHub上的代码版本，通常是团队研发中其他人更新(即推送，Push)了代码的版本， 拉取到本地，并与本地的代码进行合并。
  2. WebStorm中可以使用菜单(VCS→Git→Pull)进行拉取操作。
  3. Pull操作对话框如下：
    • 注：在进行拉取操作前，应先进行一次Commit操作。
  4. 点击对话框的“Pull”按钮，主窗体状态栏将显示Pull的进度。 如果本地文件和GitHub文件不一致，则将打开“File Merged with Conflicts”对话框：
    1. 对话框中给出了有冲突的文件列表。
    2. 右侧的“Accept Yours”和“Accept Theirs”按钮用于快速决定选定的冲突文件是使用您的本地版本还是GitHub版本。
    3. 如果不在该窗口上解决冲突，则可点击“Close”按钮。此后可以使用“VCS→Git→Resolve Conflicts”菜单 再次打开“File Merged with Conflicts”对话框。
      • 注：每次执行了Pull操作后，应该将所有的冲突均解决完后执行一次Commit操作，在开展其他的研发工作。
    4. 点击“Merge”按钮可以打开冲突解决对话框，如下图所示：
      1. 对话框以三栏式给出了选定文件的三个版本，即： 左面的本地版“Your version”、中间的结果“Result”和右面的“Changes from server”
      2. 左右两栏都以高亮的方式给出了发生冲突的代码。
      3. 点击左侧或右侧冲突代码的“✖”按钮可以忽略(Ignore)该侧的冲突代码。
      4. 点击左侧冲突代码的“《”按钮或右侧冲突代码的“》”按钮可以接受(Accept)该侧的冲突代码。
      5. 将左右两侧所有各段冲突代码都进行了Accept或Ignore操作后，应该点击右下角的Apply按钮，认可冲突解决操作。
      6. 窗口左下角给出了“Accept Left”和“Accept Right”按钮，它们用于快速地接受所有的本地或GitHub中的代码版本。

推送(Push)操作

  1. Push操作就是将本地最后一次Commit的代码上传到GitHub上的共享代码库中，以便团队中的其他成员可以拉取并合并。
  2. WebStorm中可以使用菜单(VCS→Git→Push)或快捷键(Ctrl+Shift+K)进行拉取操作。
  3. Push操作对话框的左窗格给出提交情况，右窗格示出新增的文件(绿色)和修改的文件(兰色)，如下所示：
    • 注：在进行拉取操作前，一般要先进行一次Commit操作。
  4. 点击Push按钮，主窗体状态栏将显示Push进度。 如果没有问题，则主窗体右下侧将给出成功的提示，信息如下：
    Push successful: Pushed 2 commits to origin/master.
  • 注：在Commit对话框上点击“Commit”按钮右侧的下拉箭头可以看到“Commit and Push”命令， 点击该命令系统将先执行Commit操作，然后打开“Push”对话框，以实现快速的Commit→Push操作。
